# 福建省体育招投标商业情报系统

## 🎯 项目概述

从**福建省政府采购网**（`zfcg.czt.fujian.gov.cn`）自动化抓取体育/AI体育相关招标数据，通过 **LLM 语义分析**（严禁使用正则表达式或关键词硬匹配）将原始 HTML 转化为结构化商业情报，最终输出一份可直接用于**竞争对手分析**和**商业机会捕捉**的 Excel 文件。

---

## 📐 核心约束（硬性规定）

### 1. 检索关键词
```
["体育", "赛", "赛事", "运动", "智慧体育", "智能体育", "数字体育", "智慧场馆", "体育数字化", "体育信息化", "体育大数据"]
```
策略：用宽泛词（如"赛""运动"）进行大范围搜索，再由 LLM 精筛，避免漏项。

### 2. 优先级标签（Priority）
仅以下三类标记为 `YES`：
| 类别 | 定义 | 示例 |
| :--- | :--- | :--- |
| **赛事运营** | 赛事组织、执行、宣传服务 | 足球联赛组织、龙舟比赛承办 |
| **赛事设备** | 体育器材、比赛设备、转播系统 | 体育设备升级、慢动作制作系统 |
| **AI体育** | 智慧体育、数字化、人工智能 | 智慧体育公园、AI体育测试系统 |

不属上述三类者标记为 `NO`（含体育机构采购的办公设备、车辆维保等）。

### 3. 时间分层策略（Stratum）
| 日期条件 | 标签 | 含义 |
| :--- | :--- | :--- |
| 发布日期 ≥ 2025-01-01 | `实时线索` | 当前可操作机会，高紧迫 |
| 发布日期 < 2025-01-01 | `市场情报` | 历史数据，用于价格锚点/对手分析 |

### 4. 项目状态标签（Status）
| 原始公告类型 | 标签 | 含义 |
| :--- | :--- | :--- |
| 公开招标/竞争性磋商/竞争性谈判/单一来源/询价/邀请招标 | `进行中` | 可投标机会 |
| 采购意向 | `预告中` | 提前布局线索 |
| 合同/结果/中标/成交 | `已完结` | 已完成，分析竞争对手 |

### 5. 字段完整性
**所有表头字段必须填满，严禁留空。**
- 提取到具体值 → 填入具体值
- 原页确实没有 → 填入 `原网页未披露该项`

### 6. 提取方式
**严禁使用正则表达式（Regex）和关键词硬匹配。**
必须由 LLM 逐条阅读原始 HTML 源码，通过语义理解判定优先级并提取字段。

### 7. 抓取风格
单线程、模拟人类行为（随机休眠）、严禁并发 → 避免触发政府网站 WAF/IP 封禁/验证码死循环。

### 8. 质量验收
最终交付前，随机抽取 5 个项目，进行 **原网页 ↔ 提取值** 1:1 对标核对，100% 一致视为通过。

## 🏗️ 技术架构

### 管线流程
```
┌──────────────┐    ┌──────────────────┐    ┌────────────────┐    ┌──────────────┐
│  Crawl Phase │───▶│  HTML Cleaning   │───▶│  LLM Semantic   │───▶│  Excel Export │
│  (Playwright) │    │  (BeautifulSoup) │    │  Extraction     │    │  (openpyxl)  │
└──────────────┘    └──────────────────┘    └────────────────┘    └──────────────┘
    两阶段爬取           去脚本/去样式         人工语义阅读            双Sheet输出
```

### 技术栈
| 组件 | 技术 | 用途 |
| :--- | :--- | :--- |
| 浏览器自动化 | Playwright (headless Chromium) | 模拟人类点击、翻页、详情页抓取 |
| 验证码识别 | ddddocr | 自动识别图形验证码 |
| HTML 清洗 | BeautifulSoup4 | 去除 `<script>` `<style>` 等噪声 |
| 语义提取 | LLM 直接分析 | 不写 regex，直接读 HTML 文字 |
| 数据存储 | SQLite (`llm_bidding_v2.db`) | 原始 HTML + 提取结果 |
| Excel 导出 | openpyxl | 格式化 Excel，自动筛选，高亮标注 |
| 包管理 | uv | 依赖安装与管理 |

## 🚀 使用手册

### 阶段一：数据抓取
```bash
cd C:\Users\cc\hermes-sports-bidding\llm_pipeline
python crawl.py
```
- 遍历 11 个检索词，每词爬 3 页
- **阶段1**（搜索爬取）→ 逐行点击公告列表，抓取详情页 HTML
- **阶段2**（溯源追回）→ 对合同/结果/意向公告，用项目名称反查招标公告

### 阶段二：HTML 脱水 → 数据库清理

**注意：当前实现中，`clean_v2.py` 是数据库清理脚本（清空 `pages` 和 `extracted` 表），而非 HTML 清洗器。**
实际的 HTML 清洗由导出脚本内部完成（使用 BeautifulSoup4 去除 `<script>` `<style>` 等噪声），输出 `cleaned_data.json`（精炼文本，记录数取决于爬取量）。

### 阶段三：LLM 语义提取 + Excel 导出
```bash
cd C:\Users\cc\hermes-sports-bidding
python final_export_full.py
```
- 读取 `cleaned_data.json`
- 对每条记录判定 Priority / Stratum / Status
- 对 YES 项目从 HTML 文本中提取预算、供应商、截止日期、项目描述
- 导出到桌面 Excel

## 📏 字段规格明细

| 等级 | 含义 | 适用字段 |
|:---|:---|:---|
| **🔴 硬性完整** | 必须填出具体值，不允许 `原网页未披露该项` | 序号、项目名称、时效层级、项目状态、优先级、标记理由、预算/成交金额、采购单位、公告类型、发布时间、项目描述 |

| **🔵 待完善** | 需爬虫配合保存 URL 后方可硬性完整 | 原文链接 |
| **⚪ 尽力提取** | HTML 有则填，无则 `原网页未披露该项` | 项目编号 |

### 状态感知规则详解

供应商和投标截止时间两个字段的完整性要求随项目状态变化：

| 项目状态 | 供应商要求 | 投标截止时间要求 |
|:---|:---|:---|
| **进行中**（可投标） | 允许 `原网页未披露该项`（招标阶段无供应商） | **必须完整**（HTML 第 5 节明确标注 `提交投标文件截止时间`） |
| **预告中**（采购意向） | 允许 `原网页未披露该项`（未到招标阶段） | 允许 `原网页未披露该项`（意向公告不列截止时间） |
| **已完结**（合同/结果公告） | **必须完整**（分析竞争对手） | 允许 `原网页未披露该项`（已完结公告不重复截止时间） |

> **数据依据**：审计 40 条 YES 记录的实际 HTML 源码后确认——
> - 招标公告（公开招标/竞争性磋商）的 HTML 第 5 节固定包含 `提交投标文件截止时间、开标时间和地点`
> - 合同公告和结果公告的 HTML 不包含投标截止时间字段，只包含履约期限
> - 供应商信息在结果公告和合同公告中 90%+ 可提取

| 场景 | 输出格式 | 示例 |
|:---|:---|:---|

| 有预算和中标两个金额 | 优先取中标/成交金额；预算金额作为参考 | `4,710,000元` |

### 字段提取优先级规则

当 HTML 中出现多个候选值时，LLM 必须按以下优先级选择**最相关**的值，不得随机选取。

**金额字段提取优先级**

| 优先级 | 金额类型 | 常见位置 |
|:---|:---|:---|
| 1（最高） | 中标/成交金额 | 结果公告/合同公告中的"中标金额""成交金额""合同金额" |
| 2 | 预算金额/预算价 | 招标公告中的"预算金额""采购包预算金额" |
| 3 | 最高限价 | 招标公告中的"最高限价""采购包最高限价" |
| 4 | 投标保证金 | 招标公告中的"投标保证金" |
| 5 | 其他金额 | 如分包预算、单价等 |

规则：同一 HTML 中存在多个金额时，取优先级最高的那个填入 `预算/成交金额` 字段。若无法确定类型，取**最先出现**的金额值。

**日期字段提取优先级**

| 优先级 | 日期类型 | 适用场景 |
|:---|:---|:---|
| 1（最高） | 投标截止时间/开标时间 | 公开招标/竞争性磋商等"进行中"公告 |
| 2 | 中标/成交日期 | 结果公告/中标公告 |
| 3 | 合同签订日期 | 合同公告 |
| 4 | 公告发布日期 | 所有公告类型（用于`发布时间`字段） |

规则：`投标截止时间`字段**只取**优先级 1 的值，不得将发布日期、合同签订日期等填入此字段。

**干系人字段提取优先级**

| 字段 | 第1优先 | 第2优先 | 第3优先 |
|:---|:---|:---|:---|
| 采购单位 | 采购人/招标人名称 | 业主单位名称 | 页面标题中提取的采购方 |
| 供应商 | 中标供应商 | 成交供应商 | 合同乙方/承包方 |

### 优先级判定决策树

LLM 必须按以下决策树逐层判定每条记录的优先级标签，不得跳步：

```text

      │   ├─ "体育局办公楼物业管理服务" → NO
      │   ├─ "体育局审计服务" → NO

      │
      ├─ 体育基础设施（非智慧、非设备）→ NO
```

```text
## 核心规则

### 规则1：优先级判定（逐层执行，不得跳步）
第1步：判断项目是否涉及体育/赛事/运动相关领域
第2步：如果涉及，按以下三类语义判定：
  - 赛事运营：赛事组织、执行、宣传、接待、安保、直播服务
  - 赛事设备：体育器材、比赛设备、转播系统、计时计分
  - AI体育：智慧体育、数字化、人工智能、信息化系统
第3步：边界案例处理：
  - 体育机构采购非体育专用品（空调/车辆/办公家具）→ NO
  - 名称含"体育"但内容无关（如"体育局审计服务"）→ NO
  - 纯体育基础设施（跑道翻新、座椅更换）→ NO
  - 含体育成分但主体非核心 → 按核心成分判定
→ YES 输出：对应类别 + 项目关键词（如"体育赛事运营 (足球联赛)"）
→ NO 输出："非核心业务"

### 规则2：项目状态判定
- 公开招标/竞争性磋商/竞争性谈判/单一来源/询价/邀请招标 → "进行中"
- 采购意向/采购需求预告 → "预告中"
- 合同公告/结果公告/中标公告/成交公告/更正公告/废标公告/终止公告 → "已完结"

### 规则3：时效层级判定
- 发布日期 >= 2025-01-01 → "实时线索"
- 发布日期 < 2025-01-01 → "市场情报"

### 规则4：金额提取（多候选值按优先级）
优先级：中标金额 > 预算金额 > 最高限价 > 投标保证金
- 有中标和预算两个金额时，优先取中标/成交金额
- 格式保持原文，如 "9,252,344.80元"、"560,000元"、"189.9万元"
- 无任何金额信息 → "原网页未披露该项"

### 规则5：投标截止时间提取
- 只从 HTML 中 "提交投标文件截止时间、开标时间和地点" 字段提取
- 格式：YYYY-MM-DD 或 YYYY-MM-DD HH:MM
- 合同公告/结果公告中没有此字段 → "原网页未披露该项"
- 注意：不得将"公告发布日期""合同签订日期"误填入此字段

### 规则6：供应商提取
- 已完结（合同/结果公告）→ 必须提取中标/成交供应商，不得遗漏
- 进行中（招标公告）→ "原网页未披露该项"（招标阶段无供应商信息）
- 预告中（采购意向）→ "原网页未披露该项"

### 规则7：项目描述撰写
- 50-200 字，包含项目目标、采购内容、服务范围
- 客观陈述，不得推测，不得直接复制标题
- 不得含 HTML 标签或 Markdown 格式

### 规则8：空值总则
- 所有 15 个字段必须填满，零遗漏
- 有具体值填具体值，无值按对应规则填默认值
- 不得省略任何字段或键名

### 规则9：置信度标记
提取过程中遇以下情况，在字段值**后面直接附加**对应标记（不影响原始值）：
- 字段值在 HTML 中多次出现且无法按优先级确定 → 值后加 `[需复核]`
  - 示例：`"预算/成交金额": "560,000元[需复核]"`
- 字段值从 HTML 推断得出（非原文直接出现） → 值后加 `[推断]`
  - 示例：`"投标截止时间": "2026-07-09[推断]"`
- 字段值明确且唯一 → 不加标记
- 注意：标记仅用于后处理复核，LLM 仍须填出最佳值，不得因不确定而留空
```
